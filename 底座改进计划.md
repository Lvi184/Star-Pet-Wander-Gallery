### 1.2 核心产品原则（来自 PRD 阅读说明）

所有需求以以下原则为总约束：

1. **生命感优先**：当数值效率与生命感冲突时，MVP阶段优先生命感
2. **状态一致性优先**：当自由生成与状态一致性冲突时，优先结构化事实和数据库状态
3. **可演示闭环优先**：当宏大世界观与演示闭环冲突时，优先可演示闭环
4. **命运信息隔离**：Planner 不得接收区域命运等级（PRD 10.1节）
5. **统一 Action Resolver**：PlayerController 和 AgentController 共享同一 Resolver（PRD 9.1节）
6. **回溯世界线分支**：回溯创建新 WorldlineBranch，旧分支保留为「命运回响」（PRD 10.5节）


## 三、核心玩法设计（PRD 第8-11章）

### 3.1 在线/离线无缝接管（PRD 第9章）

```
                世界一直运行
                      │
         ┌────────────┴────────────┐
         │                         │
    用户在线                    用户离线
         │                         │
    玩家直接操控                AI Agent接管
         │                         │
    一起探索                    自主探索
         │                         │
    获得相同世界状态            世界继续推进
```

**核心设计理念**：玩家和AI共用同一个角色（Character），不是"AI帮玩家挂机"，而是"玩家暂时接管这个生命"（PRD 9.1节）。

#### 在线操控模式（PRD 9.2节）

玩家可以直接控制灵兽进行：
- 🌲 **探索**：前往六域地图的各个区域
- 🌿 **采集**：收集灵药、矿石、材料
- ⚔️ **战斗**：遭遇敌人时进行战斗（MVP后置）
- 🤝 **社交**：与其他灵兽/NPC互动（MVP后置）
- 🧘 **修炼**：提升灵力和境界（MVP后置）

#### 离线自主模式（PRD 9.3节）

玩家下线后，AI Agent接管灵兽：
- Planner Agent 自主规划一天行程（**不接收命运等级**）
- Explorer Agent 自主探索世界（规则结算）
- Social Agent 自主交友互动（MVP后置）
- Narrative Agent 生成漫游日记

#### 性格冲突机制（PRD 7.2节）

灵兽有自己的性格，不会完全听从玩家：

```
玩家："以后不要去火山！"
灵兽（冒险家性格）："我是冒险家，我还是去了。"
```

**设计目的**：灵兽才像真正的生命，而不是挂机脚本。

**冲突规则**：服从倾向分数由关系信任、玩家近期介入频率、角色坚持/冒险、当前目标重要度、风险可见度和指令强度共同决定。

### 3.2 天道系统：命运、风险、死亡与回溯（PRD 第10章）

#### 命运系统（PRD 10.1-10.2节）

**核心设计**：制造玩家与灵兽的信息差。灵兽看不到抽象的区域命运等级，玩家以天道视角能看到。

```
今天命运：
青丘：普通
昆仑：幸运
东海：危险 ★★★
幽都：灾难 ★★★★★

灵兽（AI）：决定去东海探险
玩家（在线）：可以阻止它去危险区域
```

**命运等级**：普通、幸运、危险、灾难

**关键约束**：命运不等于剧本结果，危险命运只调整风险权重，角色仍可能平安归来。

#### 风险事件链（PRD 10.3节）

```
进入危险条件
    │
    ▼
规则引擎抽取事件类型
    │
    ▼
结合健康/精力/装备/天赋计算后果
    │
    ▼
产生EventRecord和CauseChain（原因链）
    │
    ▼
角色可反应：逃跑 / 求助 / 坚持 / 使用物品
    │
    ▼
再结算
    │
    ▼
normal / injured / critical / dead
```

**死亡规则**：
- 必须是多因素结果，不允许"单次随机数直接死亡"
- 至少满足：严重事件 + 角色脆弱状态 或 连续失败
- 牺牲类死亡可由高同理角色在救援事件中主动选择

**CauseChain 示例**："灾难命运+低精力+夜间进入幽都+未携带照明护符"

#### 死亡叙事（PRD 10.4节）

**三类死亡表现**：

| 类型 | 触发条件 | 表现形式 |
|------|----------|----------|
| 牺牲故事 | 因救援死亡 | 强调行为意义 |
| 传说故事 | 长期著名探险者 | 强调世界留下痕迹（墓碑） |
| 星空故事 | 普通生命 | 强调温柔告别（化作星星） |

**死亡页面流程**：讲述"发生了什么" → 说明可追溯原因 → 提供选择（回溯/阅读最后一天）

#### 天道回溯（PRD 10.5节）

**核心设计**：不是数据库回滚后重放同一随机种子，而是恢复到Checkpoint状态，创建新的WorldlineBranch，并从节点后使用新的分支种子重新演算。

```
玩家："昨天它死了。"
玩家使用"天道回溯"回到昨天
创建新的branch_id，旧分支保留为"命运回响"

第一次：去了森林 → 被狼吃了
第二次：去了河边 → 认识鸭子  
第三次：遇见商人 → 买到护符 → 活下来

玩家实际上在观察：不同人生。
```

**回溯规则**：
- 每天最多1次回溯
- 消耗天道点数
- 回溯点："昨日开始"、"重大风险前"、"死亡前最近安全点"
- 旧分支保留为"命运回响"摘要

### 4.2 核心架构约束（PRD 15.1节）

1. **CharacterState 是角色当前权威状态**
2. **所有关键变化有 EventRecord**
3. **回溯可通过 Checkpoint 和分支事件重新构建局部世界线**

### 4.3 角色架构（PRD 9.1节）

#### Character 模型（PRD 15.3节）

```python
class Character:
    """统一角色数据模型（PRD 15.3节）"""
    
    id: str
    user_id: str
    name: str
    species: str              # 物种（青丘狐、小青龙等）
    personality: dict         # 人格六维：好奇/谨慎/社交/同理/坚持/冒险（0-100）
    mood: float               # 心情 0-100
    energy: float             # 精力 0-100
    spiritual_power: float    # 灵力值（MVP保留但不主循环）
    cultivation_level: int    # 修炼境界（1-9重）
    current_region: str       # 当前区域
    inventory: List[str]      # 背包物品
    health: float             # 生命值 0-100
    status: str               # 状态：normal/injured/critical/dead
    controller_type: str      # 控制类型：player/agent/transitioning
    controller_version: int   # 控制权版本号（防并发）
    last_active: datetime
    last_player_heartbeat: datetime  # 会话心跳（PRD 9.4节）
    affinity_map: dict        # 与其他角色的好感度（MVP只读兼容）
    goals: List[str]          # 当前目标列表（短期意图/长期目标/关系承诺）
```

```python
class ActionResolver:
    """统一动作结算器（PRD 9.1节核心）"""
    
    async def resolve(self, action: ActionInput) -> EventRecord:
        """
        所有玩家和AI的动作都经过此结算器
        确保：玩家采集的珍珠和AI采集的珍珠进入同一个背包
             玩家遭遇的NPC和AI遭遇的NPC进入同一关系系统
        """
        # 1. 校验动作合法性
        # 2. 规则引擎结算结果
        # 3. 更新CharacterState
        # 4. 生成EventRecord
        # 5. 触发相关服务（记忆、叙事等）
```

### 4.5 LangGraph 工作流（PRD 第13章）

#### 4.5.1 Agent 职责矩阵（PRD 13.2节）

| Agent | 职责 | 输入 | 输出 | LLM调用频率 |
|-------|------|------|------|------------|
| **Planner** | 自主行为核心 | CharacterSummary（不含命运） | ActionIntent | 计划耗尽/重大事件/目标改变 |
| **Reflection** | 事件转化为内部意义 | 5-15个重要事件+人格+目标 | emotion_shift, lesson, memory_focus | 每日1次+重大事件额外 |
| **Narrative** | 用户可读文本生成 | EventRecord+Memory+Relationship | 漫游志文本 | 每日聚合1次 |
| **Social** | 社交对话生成 | 双方人格+关系+共享记忆 | 对话内容+SocialIntent | 相遇时2-6轮 |
| **Memory** | 记忆检索与管理 | 过滤条件+语义查询 | 记忆列表+关联 | 每次计划/叙事 |

#### 4.5.2 单角色行为图（Character Behavior Graph）

```
offline_sync → fate_check(玩家可见) → schedule_check → plan(不含命运) → [explore/collect/rest/cultivate/fight] → reflect → narrate → biography_update
```

**关键约束**：`plan` 节点的 Planner 上下文**不包含**区域命运等级（PRD 10.1节）。

#### 4.5.3 LLM/规则驱动边界

| 行为节点 | 驱动方式 | 说明 | API调用频率 |
|----------|----------|------|------------|
| `plan_step` | **LLM** (DeepSeek) | 决策下一步行动，**不接收命运等级** | 计划耗尽/重大事件 |
| `explore_step` | **规则 + LLM** | 规则决定类型，LLM丰富细节 | 关键发现时 |
| `social_step` | **LLM** | 社交对话生成 | 相遇时 |
| `reflect_step` | **LLM** | 反思总结 | 每日1次 |
| `narrate_step` | **LLM** | 日记生成（事实校验） | 每日1次 |
| `fight_step` | **规则 + LLM** | 规则计算数值，LLM生成描述 | MVP后置 |
| `schedule_check` | 规则 | 根据时间和状态决定日程 | 无 |
| `rest_step` | 规则 | 休息恢复固定数值 | 无 |
| `cultivate_step` | 规则 | 修炼增加灵力固定数值 | 无 |
| `fate_check` | 规则 | 读取命运等级（仅玩家可见） | 无 |

| Relationship | 关系 | id, char_a_id, char_b_id, type, affinity, history | 12.2 |
| Biography | 传记 | id, char_id, title, content, milestone, timestamp | 11.6 |
| NPCCharacter | NPC角色 | id, name, species, personality, location, schedule, dialogue_pool | 12.1 |

### 4.7 控制权切换机制（PRD 9.1节）

```
用户发起接管
    │
    ├─→ 校验角色未死亡 / 未被其他会话占用
    │
    ├─→ controller_type = transitioning（中间态防并发）
    │
    ├─→ 等待当前AI动作完成或安全取消
    │
    ├─→ 写入 ControlHandoverEvent
    │
    ├─→ controller_version += 1（版本号防并发）
    │
    ├─→ controller_type = player
    │
    ├─→ 推送最新 CharacterState
    │
    └─→ 玩家操作进入统一 Action Resolver
```

**会话心跳（PRD 9.4节）**：
- 客户端维持会话心跳，服务端记录 `last_player_heartbeat`
- 超过阈值且无有效动作时，系统进入可交接状态
- 玩家主动点击"让它自己去吧"可立即发起交接
- Demo模式提供"模拟离线"按钮

## 六、MVP Demo 边界（PRD 第16章）

**MVP 战略目标（PRD 16.1节）**：让评委或测试用户在10分钟内理解并相信三件事：
1. 这个世界在玩家离线后继续运行
2. AI和玩家控制的是同一个生命而不是两个模式
3. 命运与回溯能够改变故事分支

**MVP Demo 演示脚本（PRD 16.3节）**：

```
1. 打开项目：首页直接显示灵儿的《今日漫游志》，指出"这些事件发生在用户离线期间"
2. 进入六域地图：看到灵儿当前在灵墟，同时东海命运为"危险"
3. 点击"与它同行"：服务端完成ControlHandover，地图状态不重置
4. 玩家移动到青丘并采集青丘草，背包和事件流立即更新
5. 点击"让它自己去吧"或Demo模拟离线，Agent读取人格与目标后形成计划
6. 冒险型灵儿自主前往东海。系统明确：Planner看不到"危险"命运，玩家能看到
7. 规则引擎触发风险链，灵儿受伤；漫游志增加危险事件（显示CauseChain）
8. 继续演示可控死亡分支，生成"最后一天"而非Game Over
9. 玩家使用天道回溯返回风险前Checkpoint，创建新branch_id
10. 重新演化：灵儿可能遇到商人、得到护符或改变路线，展示世界线不同
11. 结束时打开Agent观察台，展示Planner意图、规则结算和Narrative引用事件，证明不是随机写故事
```

## 八、比赛亮点总结（PRD 第1章、第16章）

| 亮点 | 描述 | 技术支撑 |
|------|------|----------|
| **AI数字生命** | 每只灵兽都是拥有自主意识的数字生命 | LangGraph + DeepSeek + 人格六维 |
| **在线/离线无缝接管** | 玩家在线直接操控，离线AI接管，共享同一Action Resolver | PlayerController/AgentController/统一Resolver |
| **持续运行世界** | 世界时间独立运行，规则Tick与叙事Tick分离 | APScheduler + WorldTimeEngine |
| **命运信息隔离** | 玩家可见命运，AI不可见，制造守护意义 | DestinySystem + Planner上下文隔离 |
| **天道回溯** | 回档创建新世界线，旧分支保留为命运回响 | WorldlineBranch + branch_id |
| **可追溯风险链** | 死亡必须有多因素CauseChain，可解释 | CauseChain + EventRecord |
| **AI生成故事** | 死亡是叙事节点，而非固定剧情 | Narrative Agent + 事实校验 |
| **Agent观察台** | 向评委证明系统具备可审计的Agent链路 | Planner意图 + 规则结算 + 事件引用 |
