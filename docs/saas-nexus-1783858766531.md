#### 2.2.4 世界引擎落地机制

当前项目基于 FastAPI + SQLite，无后台调度器。为实现「世界独立于用户持续运行」，需引入后台任务调度机制。

**技术选型**：

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **APScheduler** | 轻量、易于集成、支持多种触发器 | 单进程、不支持分布式 | Demo/比赛场景 |
| **asyncio background task** | 原生支持、与FastAPI无缝集成 | 进程退出即停止 | 开发阶段 |
| **Celery + Redis** | 分布式、支持定时任务 | 依赖多、部署复杂 | 生产环境 |

**推荐方案**：APScheduler（适合比赛Demo）

**FastAPI Lifespan 集成**：


python
# backend/main.py
from apscheduler.schedulers.asyncio import AsyncIOScheduler
from services.world_time import WorldTimeEngine

scheduler = AsyncIOScheduler()

@app.on_event("startup")
async def startup():
    """启动世界引擎"""
    world_time_engine = WorldTimeEngine()
    
    scheduler.add_job(
        world_time_engine.tick,
        trigger="interval",
        seconds=int(os.getenv("WORLD_TICK_INTERVAL", 3600)),
        id="world_tick",
        name="世界时间Tick",
    )
    scheduler.start()

@app.on_event("shutdown")
async def shutdown():
    """停止世界引擎"""
    scheduler.shutdown()

## 二、核心玩法设计

### 2.1 在线/离线无缝接管


世界一直运行
                      │
         ┌────────────┴────────────┐
         │                         │
    用户在线                    用户离线
         │                         │
    玩家直接操控                AI Agent接管
         │                         │
    一起探索                    自主探索
         │                         │
    获得相同世界状态            世界继续推进


**核心设计理念**：玩家和AI共用同一个角色，不是"AI帮玩家挂机"，而是"玩家暂时接管这个生命"。

#### 在线操控模式

玩家可以直接控制灵兽进行：
- 🌲 **探索**：前往六域地图的各个区域
- 🌿 **采集**：收集灵药、矿石、材料
- ⚔️ **战斗**：遭遇敌人时进行战斗
- 🤝 **社交**：与其他灵兽/NPC互动
- 🧘 **修炼**：提升灵力和境界

#### 离线自主模式

玩家下线后，AI Agent接管灵兽：
- Planner Agent 自主规划一天行程
- Explorer Agent 自主探索世界
- Social Agent 自主交友互动
- Narrative Agent 生成漫游日记

#### 性格冲突机制（关键设计）

灵兽有自己的性格，不会完全听从玩家：


玩家："以后不要去火山！"
灵兽（冒险家性格）："我是冒险家，我还是去了。"


**设计目的**：灵兽才像真正的生命，而不是挂机脚本。

### 2.2 风险与死亡机制

**死亡不是失败，而是故事的一部分**。

#### 风险事件类型

| 事件 | 描述 | 后果 |
|------|------|------|
| 受伤 | 在危险区域遭遇攻击 | 灵力下降，需休息恢复 |
| 被骗 | 遇到奸诈NPC | 丢失部分物品 |
| 迷路 | 在复杂地形迷失 | 多花费时间和精力 |
| 重伤 | 遭遇强力敌人 | 需要玩家救援或AI自救 |
| 死亡 | 伤势过重或遭遇致命危险 | 触发死亡叙事 |

#### 死亡叙事设计

**方案一：牺牲故事**

为了救朋友牺牲了
→ 玩家回来看到："这是它最后一天。今天救了一只小鹿。最终倒在雪地。"


**方案二：成为传说**

死亡 → 留下墓碑 → 其他灵兽经过会读它的故事 → 世界留下痕迹


**方案三：进入星空**

死亡 → 化作星星 → 照亮别人 → 其他灵兽看到流星会说"听说那是前辈"


### 2.3 回档机制（天道回溯）

**不是无限撤销，而是回到命运节点重新演化**。


玩家："昨天它死了。"
玩家使用"天道回溯"回到昨天
世界重新演算，但不会完全一样

第一次：去了森林 → 被狼吃了
第二次：去了河边 → 认识鸭子
第三次：遇见商人 → 买到护符 → 活下来

玩家实际上在观察：不同人生。


**世界观包装**：

| 名称 | 描述 |
|------|------|
| 时间沙漏 | 使用时间沙漏，回到昨天 |
| 星之记忆 | 翻开星之记忆，重新开始那一天 |
| 命运回响 | 命运重新演算 |

**回档规则**：
- 每天限用1次
- 回档后世界线会发生变化（NPC关系、事件、掉落等重新演化）
- 回档消耗"天道点数"（可通过在线时长或成就获得）

### 2.4 命运系统

世界每天随机生成命运等级，灵兽不知道但玩家可以看到：


今天命运：
青丘：普通
昆仑：幸运
东海：危险 ★★★
幽都：灾难 ★★★★★

### 4.4 LangGraph 工作流

offline_sync → fate_check → schedule_check → plan → [explore/social/rest/cultivate/fight] → reflect → narrate → biography_update

#### 4.4.2 LLM/规则驱动边界

| 行为节点 | 驱动方式 | 说明 | API调用频率 |
|----------|----------|------|------------|
| plan_step | **LLM** (DeepSeek) | 决策下一步行动 | 每行为一次 |
| explore_step | **规则 + LLM** | 规则决定类型，LLM丰富细节 | 可选 |
| social_step | **LLM** | 社交对话生成 | 每社交一次 |
| reflect_step | **LLM** | 反思总结 | 每轮结束 |
| narrate_step | **LLM** | 日记生成 | 每轮结束 |
| fight_step | **规则 + LLM** | 规则计算数值，LLM生成描述 | 每战斗一次 |
| schedule_check | 规则 | 根据时间和状态决定日程 | 无 |
| rest_step | 规则 | 休息恢复固定数值 | 无 |
| cultivate_step | 规则 | 修炼增加灵力固定数值 | 无 |
| fate_check | 规则 | 读取命运等级 | 无 |

1.3 项目背景
近年来，大语言模型（LLM）与 Agent 技术快速发展，使游戏角色具备长期记忆、自主规划、情绪变化和自然交流成为可能。
然而，目前绝大多数 AI 宠物产品仍停留在以下阶段：


聊天机器人


桌面宠物


陪伴APP


电子宠物


AI NPC


这些产品虽然能够进行一定程度的交互，但并没有真正“活着”。
它们普遍存在几个核心问题：
第一，世界不会持续运行。
只有当玩家打开应用时，宠物才开始行动；玩家关闭应用后，整个世界立即停止。
第二，角色没有真正的目标。
大多数 AI 宠物只能等待玩家发出指令，没有主动探索、成长、建立关系或形成长期故事的能力。
第三，缺乏连续叙事。
用户每天看到的是一次次孤立的互动，而不是一个不断演化的人生故事。
因此，玩家很难产生真正的情感连接，也无法形成长期留存。

1.4 产品愿景
《星宠漫游馆（山海灵境世界观）》希望重新定义 AI 宠物。
我们的目标不是创造一只能够聊天的宠物，而是创造一个持续运行的世界，以及生活在其中的数字生命。
在这个世界中：


世界永不停息。


每只灵兽拥有独立人格。


每只灵兽拥有长期记忆。


每只灵兽能够主动制定计划。


每只灵兽能够结交朋友。


每只灵兽能够探索世界。


每只灵兽能够面对危险。


每只灵兽都会成长。


每只灵兽都会老去。


每只灵兽也可能死亡。

关于《我的世界》
如果你说的是：

"像《我的世界》一样可以自由探索"

这是一个很好的参考。
如果你说的是：

"直接把《我的世界》的官方世界放进来"

那就不建议，因为官方资源仍然受版权保护。
不过有很多**Minecraft 风格（Voxel）**的开源项目，你可以借鉴它们的地图格式和生成方式，而不用复制官方内容。

我觉得你可以再大胆一点
你的项目最大的优势，其实不是地图，而是世界可以持续存在。
可以把它设计成：

世界接口（World Adapter）

任何符合规范的地图都能接入。
例如定义一个统一接口：
Plain textMap├── 地形├── 建筑├── 可交互物体├── NPC 出生点├── 资源点└── 世界规则
AI Agent 永远调用的是：
代码Move()Observe()Talk()Harvest()Build()Sleep()
而不是关心地图来自哪里。
这样未来甚至可以支持：

AI 自动生成的新大陆


它们都会成为《星宠漫游馆》的一个"世界分馆"。
从长期来看，我认为你的核心产品不是某一张地图，而是一个"AI 世界操作系统"——任何经过授权、符合规范的世界，都可以接入并拥有会生活、会成长、会留下历史的居民。这会比围绕某一个 IP（如《塞尔达》或《我的世界》）构建，更有生命力，也更容易持续发展。
