#### 2.2.4 世界引擎落地机制

当前项目基于 FastAPI + SQLite，无后台调度器。为实现「世界独立于用户持续运行」，需引入后台任务调度机制。

**技术选型**：

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **APScheduler** | 轻量、易于集成、支持多种触发器 | 单进程、不支持分布式 | Demo/比赛场景 |
| **asyncio background task** | 原生支持、与FastAPI无缝集成 | 进程退出即停止 | 开发阶段 |
| **Celery + Redis** | 分布式、支持定时任务 | 依赖多、部署复杂 | 生产环境 |

**推荐方案**：APScheduler（适合比赛Demo）

**FastAPI Lifespan 集成**：


python
# backend/main.py
from apscheduler.schedulers.asyncio import AsyncIOScheduler
from services.world_time import WorldTimeEngine

scheduler = AsyncIOScheduler()

@app.on_event("startup")
async def startup():
    """启动世界引擎"""
    world_time_engine = WorldTimeEngine()
    
    scheduler.add_job(
        world_time_engine.tick,
        trigger="interval",
        seconds=int(os.getenv("WORLD_TICK_INTERVAL", 3600)),
        id="world_tick",
        name="世界时间Tick",
    )
    scheduler.start()

@app.on_event("shutdown")
async def shutdown():
    """停止世界引擎"""
    scheduler.shutdown()

## 七、风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|----------|
| 世界持续运行导致资源消耗过高 | 服务器负载大 | 采用定时任务而非实时循环，限制并发数 |
| 多宠物社交计算复杂度高 | 响应延迟 | 社交检测采用区域索引，异步执行 |
| LLM调用成本 | API费用高 | 缓存常用结果，限制每日调用次数 |
| 数据一致性 | 并发写入冲突 | 使用数据库事务和乐观锁 |
